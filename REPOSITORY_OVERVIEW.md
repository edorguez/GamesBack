# GamesBack — Repository Overview & Architecture Summary

This document summarizes the design, architecture, technologies, and notable patterns used in the GamesBack repository (https://github.com/edorguez/GamesBack). It was generated by inspecting the repository structure and representative files.

## What this is
A .NET 7 REST API that implements Clean Architecture and Domain-Driven Design to manage "games" (CRUD). It exposes Carter-based HTTP endpoints that dispatch MediatR commands/queries (CQRS) implemented in the Application layer, persists data with EF Core + PostgreSQL, maps DTOs with Mapster, and uses FluentValidation and FluentResults for validation and error handling.

## Stack
- **Language(s):** C# (100%)
- **Framework / runtime:** .NET 7 (ASP.NET Core)
- **Notable libraries:**
  - MediatR — mediator and CQRS pipeline
  - EF Core — ORM (PostgreSQL as the backing database)
  - Mapster / MapsterMapper — object mapping between DTOs and domain
  - FluentValidation — request and command validation
  - FluentResults — typed operation results / error handling
  - Carter — modular endpoint routing for minimal APIs

## How it's organized
Annotated structure (high-level):

```text
  GamesBack.Application/         Application layer (CQRS handlers, validators)
    DependencyInjection.cs       registers MediatR, validators, mapping
    Games/
      Commands/                  Create/Update/Delete commands + handlers + validators
      Queries/                   GetGame / GetAllGame queries + handlers
  GamesBack.Contracts/           DTOs / request/response contracts (GameRequest/Response)
  GamesBack.Domain/              Domain entities, value objects, domain interfaces
  GamesBack.Infrastructure/      EF Core DbContext, repositories, data access
```

How it fits together:
- Carter modules in GamesBack.API expose routes and map incoming requests to Contracts (DTOs).
- DTOs are mapped to MediatR Commands/Queries via Mapster/IMapper and sent to the application layer.
- MediatR handlers in GamesBack.Application implement CQRS: commands change state, queries read state.
- Handlers rely on repository/unit-of-work abstractions implemented by GamesBack.Infrastructure, which uses EF Core to persist to PostgreSQL.
- FluentValidation validates inputs; FluentResults is used by handlers to return operation success/failure with typed errors (e.g., NotFoundError), and the API inspects those results to return proper HTTP statuses.

## How to run it (shortest path)
1. Install .NET 7 SDK and have a PostgreSQL instance running.
2. Update the connection string in src/GamesBack.API/appsettings.json (or use environment variables).
3. From the repository root:

```bash
dotnet restore
dotnet build
dotnet run --project src/GamesBack.API
```

Notes:
- The project uses EF Core migrations if present; apply them to your PostgreSQL database as needed (e.g., `dotnet ef database update` from the Infrastructure project if migrations are included).
- Environment configuration (connection strings, secrets) is required before the application can connect to the database.

## Practical notes & conventions observed
- Clean Architecture: separation into API / Application / Domain / Infrastructure projects.
- CQRS pattern via MediatR: commands/queries and handlers live in the Application layer.
- Error/result handling implemented with FluentResults — handlers return Result<T> and API translates errors into HTTP responses.
- Mapping is centralized via Mapster/IMapper usage (DTOs <-> domain models).
- Carter is used rather than controllers to define modular route handlers (see Games.cs for examples).
